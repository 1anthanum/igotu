# CLINICAL_INVARIANTS.md — 临床不变量清单

> 本文件列出 IGOTU 中所有**临床敏感**的代码与内容位置。
> 这些位置的变更不属于"工程问题"，而属于"临床决策"。
> 所有 AI 编码助手在阅读本仓库时必须先读 [CLAUDE.md](./CLAUDE.md) 与本文件。

---

## 0. 责任声明

IGOTU 不是医疗设备，不替代专业诊断与治疗。但作为面向心理困扰用户的工具，
本项目的临床内容必须遵循：

1. **最小伤害原则**：宁可保守，不可激进。
2. **可追溯原则**：所有临床内容的来源、版本、审查历史必须可查。
3. **审查前置原则**：临床内容的修改在合并前必须经过临床审查（哪怕审查者是兼职/顾问）。

未完成临床审查的修改一律不得部署到生产环境。

---

## 1. 不变量分级

| 级别 | 含义 | 修改许可 |
|------|------|---------|
| **A1 — 绝对不变** | 标准化量表内容、监管规定的内容 | 不可修改（除非外部标准更新） |
| **A2 — 临床可变** | 设计原则下的临床话术、流程 | 仅经临床审查可修改 |
| **B — 工程契约** | 反映临床语义的数据结构 | 工程师可改但必须保持语义一致 |

### 2.1 PHQ-9 量表

**位置**：
- `frontend/src/components/toolbox/PHQ9Assessment.vue`
- `backend/src/routes/phq9.ts`
- `backend/src/services/AnalyticsService.ts`（如涉及评分逻辑）

**约束**：
- 9 道题目数量不可变
- 每题选项 0-3 分制不可变
- 总分范围 0-27 不可变
- Severity 分级阈值（基于 Kroenke et al. 2001）：
  - 0-4: minimal / 无
  - 5-9: mild / 轻度
  - 10-14: moderate / 中度
  - 15-19: moderately severe / 中重度
  - 20-27: severe / 重度
- 题目原文不得意译、改写、"使其更友好"——必须保持与权威翻译一致

**修改许可**：仅当 APA 或权威机构发布修订版时，且经临床顾问书面确认。

## 3. A2 级不变量（临床可变）

### 3.1 安抚话术与"零内疚"原则

**位置**：`backend/src/utils/constants.ts`，`ENCOURAGEMENT` 对象。

**核心设计原则**（来自代码注释 `// NEVER guilt-inducing`）：

- 不假设用户"应该"达到某种节奏（"今天又没记录"是错的，"我们随时在这里"是对的）
- 0 记录情况必须有**正向**回应（不可空、不可批评）
- 不使用"加油""坚持"等可能让深度抑郁用户感到压力的词
- 不与其他用户横向比较
- 数字成就用"算数""都不容易"等中性肯定，避免"好棒""超棒"等过度鼓动

**修改许可**：可由懂得抑郁话术设计的人修改，但每次修改需在 PR 中说明：被替换的文案是什么、新文案是什么、为什么后者更符合零内疚原则。

---

### 3.2 OfflineChat 的非危机分支回应文本

**位置**：`backend/src/services/OfflineChat.ts`，priority < 90 的规则的 `response.text` 与 `choices`。

**核心设计原则**（来自代码注释 `不安慰，要分析；不简化，要深入`）：

- 不空洞共情（避免"我懂你"这类表层话术）
- 提供心理学/认知层面的具体框架（如焦虑的"如果……怎么办"循环、被动意念的定义）
- 选项 A/B/C/D 必须给用户**真实的多种解读路径**，不是引导性单选

**修改许可**：可修改文本，但需保持上述风格特征。修改必须在 PR 中附"对比示例"。

---

### 3.3 认知重构（Cognitive Restructuring）流程

**位置**：`frontend/src/components/toolbox/CognitiveRestructuring.vue`

**约束**：
- 步骤顺序遵循 CBT 标准（识别想法 → 识别情绪 → 识别认知扭曲 → 找支持证据 → 找反证 → 形成平衡想法 → 重新评估强度）
- "intensity_before" 与 "intensity_after" 必须分别采集（这是 CBT 的核心反馈机制）

**修改许可**：步骤可优化呈现，但顺序与采集点不可省略。

## 4. B 级不变量（工程契约）

下列工程结构反映了上述临床语义，工程师可以重构实现，但**语义必须保持**：

### 4.1 数据库 CHECK 约束

```
mood_score        ∈ [1, 5]
energy_level      ∈ [1, 5]
phq9.score        ∈ [0, 27]
phq9.functional_impact ∈ [0, 3]
intensity_before  ∈ [1, 10]
intensity_after   ∈ [1, 10]
chat_messages.role ∈ {'user', 'assistant'}
exercise_logs.type ∈ {'breathing', 'grounding'}
```

**修改许可**：扩大范围或新增枚举值需迁移；缩小范围或删除值会导致历史数据失效，禁止。

### 4.2 Achievement 分类枚举

**位置**：`backend/src/utils/constants.ts`，`CATEGORIES` 数组。

```
self-care | movement | social | nutrition | rest | hygiene | productivity | custom
```

**修改许可**：增加新分类需补 i18n、补 `category_milestone` 文案、补迁移脚本（迁移历史数据无需变更，新分类不影响存量）。删除分类禁止。

### 4.3 Demo 模式路由白名单

**位置**：`frontend/src/router/index.ts`，`DEMO_ALLOWED_ROUTES`。

**约束**：未登录用户在 demo 模式下只能访问该集合内的路由。任何涉及个人数据持久化的路由不得加入此集合。

**修改许可**：增加路由前必须确认该路由不写入用户数据库；删除路由可。

## 5. 变更流程

### A1 级
1. 由临床顾问发起或确认
2. 在 PR 中附**外部权威来源**（DSM-5、APA、相关学术文献、政府公告）
3. 至少一名工程师 + 一名临床顾问双签
4. commit message 使用 `clinical: ` 前缀，body 中引用来源链接

### A2 级
1. 修改者在 PR 中附**对比说明**（旧版 / 新版 / 修改理由 / 哪条原则被加强）
2. 至少一名熟悉抑郁话术设计的人审查
3. commit message 使用 `clinical: ` 前缀

### B 级
1. 工程师可在常规 PR 流程中处理
2. 但必须在 PR 描述中**显式说明**："本次修改不改变 X 的临床语义"
3. 包含相应的回归测试

## 6. 临床审查检查表

每次 A 类修改的 reviewer 应回答：

- [ ] 修改后的内容是否仍然遵守"最小伤害原则"？
- [ ] 修改是否引入了任何"应该""必须""加油""坚持"等隐含期待？
- [ ] 修改是否减少了用户在危机时刻的可见求助路径？
- [ ] 修改是否绕过了既有的危机识别 → 危机回应链路？
- [ ] 修改后的文案在抑郁状态下读起来是温和的还是有压力的？（请实际读一遍）
- [ ] 是否补充了对应的回归测试？

任何一项答否，PR 不可合并。

## 7. 紧急情形

如果发现生产环境中的临床内容存在风险（错误的热线号码、被破坏的危机识别、被误改的安抚话术），
**优先级高于所有其他工程任务**。处理顺序：

1. 立即回滚到上一已知安全版本
2. 在 issue 中标记 `clinical-incident` 标签
3. 复盘时记录：发现时间、影响时间窗、可能受影响的会话数量、根因
4. 复盘结果归档到 `docs/incidents/`（待该目录建立后）
