# IGOTU 产品愿景：渐进式三层交互模型

> 核心原则：**最低启动门槛 × 最小认知负荷 × 渐进深入**
>
> 用户在任何情绪状态下都能使用产品，交互深度由状态自然决定，不由 UI 强制。

---

## 一、设计哲学

### 竞品分析与差异化定位

| 维度 | Bearable / Daylio | 传统心理 App | IGOTU |
|------|-------------------|-------------|-------|
| 启动门槛 | 注册 → 选类别 → 手动记录 | 注册 → 问卷 → 文本交互 | 打开即用，零文本可用 |
| 危机支持 | 无（纯记录工具） | 热线号码 / 文本危机干预 | 身体感官通道（触觉/听觉） |
| 情绪低谷时 | 仍需手动输入 | 认知行为练习（需文字） | 被动陪伴 + 零文本宣泄 |
| 数据汇报 | 导出 CSV / PDF | 无标准化输出 | 自动摘要 → 治疗师可读 |

### IGOTU 的独特主张

**"颤抖着也能用"** — 当用户处于最差状态时（恐慌发作、严重抑郁、自杀意念边缘），传统 App 的文本交互和认知练习形同废纸。IGOTU 的底层必须是**身体级别的交互通道**：不需要阅读，不需要思考，只需要触碰和感受。

这并非降低标准，而是承认一个临床事实：**认知功能在危机状态下会急剧下降**。一个真正的心理健康工具必须在认知最低点仍然可达。

## 二、三层交互模型

```
┌─────────────────────────────────────────────┐
│          Layer 3: 汇报与自省                  │
│   治疗师摘要 · 自我回顾 · 模式识别             │
│   ─────────── 需要认知 + 反思 ───────────     │
├─────────────────────────────────────────────┤
│          Layer 2: 记忆与关怀                  │
│   次日签到 · 危机记忆 · 温和跟进               │
│   ─────────── 需要最少的认知参与 ───────────   │
├─────────────────────────────────────────────┤
│          Layer 1: 感官通道                    │
│   沙袋 · 音乐 · 呼吸震动 · 触觉反馈           │
│   ─────────── 零认知，纯身体 ───────────      │
└─────────────────────────────────────────────┘

        ↑ 用户状态改善时自然上浮
        ↓ 状态恶化时自然下沉
```

### 状态驱动的层级选择

层级不是菜单选项。用户不"选择"进入某一层——系统根据情绪分数和行为信号自动调整可用性。

| 情绪分数 | 默认层级 | 可用层级 | 页面行为 |
|---------|---------|---------|---------|
| 1 | Layer 1 | 1 | 全屏感官模式，Sanctuary 扩展 |
| 2 | Layer 1-2 | 1, 2 | Sanctuary + 简化工具 + 次日关怀 |
| 3 | Layer 2 | 1, 2, 3 | 正常首页 + 所有功能可达 |
| 4-5 | Layer 2-3 | 1, 2, 3 | 正常首页 + 汇报/分析突出 |

## 三、Layer 1：感官通道（Crisis Mode）

### 设计约束

- **零文字交互**：所有操作通过触碰、滑动、长按完成
- **视觉极简**：深色背景 + 单个焦点元素 + 柔和动效
- **声音作为治疗通道**：不只是背景，是主动的感官输出
- **无决策负担**：不呈现选项列表，自动提供最合适的单一工具

### 1.1 Punch Bag（宣泄沙袋）

**核心交互**：用户反复点击/长按屏幕中的"沙袋"元素。每次击打产生物理反馈（视觉冲击波 + 可选震动），累积击打逐渐改变视觉状态（从紧张到舒缓）。

**交互流程**：

```
进入 → 屏幕中央出现柔和脉冲圆 → 用户点击/长按
                                       ↓
              短促触觉震动 (Vibration API)
              + 冲击波扩散动画
              + 累积计数器（不显示数字，用视觉表达）
                                       ↓
              30次后 → 色调从紫/靛(紧张)渐变为青绿(平静)
              + 脉冲频率自动放缓
              + 底部出现极淡的"呼吸"提示
                                       ↓
              用户停止击打 > 5s → 自动过渡到呼吸引导
```

**视觉设计**：
- 中央圆形使用 `var(--mood-accent)` 带脉冲 glow
- 击打时圆形缩放 + 径向波纹
- 累积效果：背景从 `mood 1 palette` 渐变到 `mood 3 palette`
- 完成后无文字表扬——只有视觉上的安宁

**技术实现**：
- 新组件：`components/crisis/PunchBag.vue`
- 震动：`navigator.vibrate()` (短脉冲 50ms)
- 动画：CSS `@keyframes` + `requestAnimationFrame` 控制状态
- 累积状态：组件内 ref，不持久化

**伦理注意**：沙袋是宣泄工具，不是暴力引导。视觉语言必须是"释放压力"而非"攻击"。冲击波是能量扩散，不是破坏。


### 1.2 音乐/声景通道

**核心交互**：一键播放预设声景，无需浏览或选择。

**交互流程**：

```
Sanctuary 模式下 → 底部出现声音图标（波浪 icon）
                                       ↓
              点击 → 自动播放当前情绪对应声景
              Mood 1: 轻柔白噪声 / 雨声
              Mood 2: 溪流 / 低频环境音
                                       ↓
              再次点击 → 暂停
              长按 → 切换到下一个声景（不显示列表）
```

**声景映射**：

| 情绪 | 声景类型 | 设计意图 |
|-----|---------|---------|
| 1 | 雨打窗 / 深夜虫鸣 | 包裹感，"你不孤独" |
| 2 | 溪流 / 远处雷声 | 渐变感，"天快亮了" |
| 3 | 森林 / 鸟鸣 | 平静与自然连接 |

**技术实现**：
- 新组件：`components/crisis/SoundScape.vue`
- 音频：Web Audio API (`AudioContext`) 或 `<audio>` + 预加载
- 音频资源：`public/audio/` 下 3-5 个循环 mp3（每个 < 500KB，30s 循环）
- 与 SanctuaryView 整合：作为底部浮动控件

**风险评估**：音频自动播放在移动端受限（需用户首次交互触发 AudioContext）。需要在 Sanctuary 首次触碰时解锁音频上下文。


### 1.3 呼吸震动引导

**当前状态**：已有 `BreathingExercise.vue`，但基于文字提示 + 视觉圆形。

**扩展方向**：
- 添加震动节律（吸气时无震动，呼气时长震动 200ms × 3）
- 在 Layer 1 模式下自动启动（无需点击"开始"按钮）
- 去掉文字计时器，只保留圆形的膨胀/收缩

## 四、Layer 2：记忆与关怀

### 设计约束

- **主动跟进，但不侵入**：系统在用户回到正常状态后温和地提起过去的低谷
- **记忆是关怀的基础**：记住用户说过什么、经历了什么，才能在次日提供有意义的跟进
- **敏感表达检测**：识别对话中的危机信号，但处理方式是温暖的跟进而非冷冰冰的警报

### 4.1 危机时刻记忆系统

**核心逻辑**：当用户在 Layer 1 或低情绪状态下使用了产品（Punch Bag、Sanctuary、危机工具），系统记录一个"危机时刻标记"。

**数据模型**：

```typescript
interface CrisisMarker {
  id: string;
  timestamp: number;
  /** 触发时的情绪分数 */
  moodScore: number;
  /** 使用了哪些工具 */
  toolsUsed: ('punchBag' | 'soundscape' | 'breathing' | 'sanctuary' | 'grounding')[];
  /** 持续时长（秒）— 用户在 Layer 1 停留了多久 */
  duration: number;
  /** 是否包含敏感表达（如对话中检测到） */
  hasSensitiveExpression: boolean;
  /** 次日是否已跟进 */
  followedUp: boolean;
  /** 跟进时的情绪分数 */
  followUpMood?: number;
}
```

**存储**：`igotu_crisis_markers` (localStorage)，保留最近 90 天。

**注意**：此系统不存储对话内容本身，只存储元数据标记。对话内容留在后端对话系统中。


### 4.2 敏感表达识别

**范围界定**：这不是临床诊断系统。IGOTU 不做自杀风险评估——那是专业临床工作者的职责。

**IGOTU 做的是**：在对话中检测到可能的求助信号时，标记该时刻，并在次日温和地关心用户状态。

**检测策略**（后端 + 前端协作）：

```
用户对话 → 后端 AI 分析 → 返回 meta.sensitivity_flag
                                       ↓
          前端收到 flag → 创建 CrisisMarker
          + 当前对话不中断、不弹窗、不打断
                                       ↓
          次日签到时 → 温和询问
```

**前端职责**：
- 接收后端返回的 `sensitivity_flag` 字段
- 创建 `CrisisMarker`，`hasSensitiveExpression: true`
- 不在当前对话中做任何额外处理（不弹窗、不打断、不显示资源链接）

**次日跟进措辞**（i18n key: `followUp.sensitive`）：
- "昨天你经历了一些不容易的时刻。今天感觉怎么样？"
- 不提及具体内容，不重复用户的话
- 提供简化的情绪选择（3 级：还是很难 / 好一些了 / 没事了）

**伦理红线**：
- 不存储用户的具体敏感表达内容
- 不用机器判断替代人类临床判断
- 不在危机时刻弹出干扰性的资源链接（这会增加认知负担）
- 如果后端检测到紧急状况，由后端决定是否在对话中直接提供危机资源


### 4.3 次日签到系统

**当前状态**：已有 `useMoodCheckIn.ts`，执行每日情绪签到弹窗。

**扩展设计**：

```
用户打开 App
    ↓
检查 CrisisMarker：昨天是否有 followedUp === false 的标记？
    ↓
┌─ 有 ──→ 温和签到模式
│         "昨天你待在这里一会儿。今天感觉怎么样？"
│         [emoji 1-5 选择，无文字]
│         选择后 → marker.followedUp = true
│         如果 mood ≤ 2 → 不追问，显示 "我在这里"
│         如果 mood ≥ 3 → "很高兴你好一些了" + 正常首页
│
└─ 无 ──→ 正常签到
          "今天感觉怎么样？"
          [emoji 1-5 选择]
```

**关键设计决策**：
- 签到界面不显示"昨天你说了XXX"——不重复潜在的创伤性内容
- 如果用户连续 3 天 mood ≤ 2，不加码询问，而是默默将 Layer 3 的摘要标记为"建议尽快与治疗师沟通"

## 五、Layer 3：汇报与自省

### 设计约束

- **自动生成，用户确认**：摘要由系统生成，用户可编辑后分享
- **面向两个受众**：用户自己（自省）和治疗师（临床参考）
- **隐私先行**：所有导出都在用户主动操作下完成，不自动发送

### 5.1 发展摘要

**核心功能**：自动汇总过去 N 天的情绪变化、工具使用、危机时刻、和对话关键主题。

**摘要结构**：

```
┌─────────────────────────────────────┐
│  📊 最近 7 天概览                     │
│                                     │
│  情绪趋势: [迷你火花线]               │
│  平均: 2.8 → 3.4 (↑)               │
│  波动: 中等                          │
│                                     │
│  🔧 使用工具                         │
│  呼吸练习 ×3 · 沙袋 ×1 · 对话 ×5    │
│                                     │
│  ⚡ 值得关注                         │
│  · 周三有一段困难时刻（停留 15 分钟）  │
│  · 周五情绪明显好转                   │
│                                     │
│  💬 对话主题                         │
│  睡眠 · 工作压力 · 人际关系           │
│                                     │
│  [分享给治疗师]  [保存为 PDF]         │
└─────────────────────────────────────┘
```

**数据来源**：

| 数据项 | 来源 | 当前可用？ |
|-------|------|-----------|
| 情绪趋势 | `igotu_mood_log` | ✅ 已有 |
| 工具使用 | `useExerciseTracker` | ✅ 已有 |
| 危机时刻 | `igotu_crisis_markers` | ✅ 已有 (useCrisisTracker) |
| 对话主题 | 后端 AI 摘要 API | ❌ 待建 (需后端支持) |
| 签到记录 | `useMoodCheckIn` | ✅ 已有 |

### 5.2 治疗师桥接增强

**当前状态**：已有 `TherapyBridge.vue`，但功能待明确。

**扩展方向**：
- 将发展摘要格式化为治疗师友好的格式（简洁、客观、无情绪化语言）
- 支持导出为 PDF 或可分享链接
- 用户可以在分享前编辑/删除任何条目
- 标注"用户主动分享"——不是系统泄露

## 八、风险与约束

### 技术风险

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| Vibration API 兼容性 | iOS 不支持 `navigator.vibrate()` | 降级为纯视觉反馈；不依赖震动作为核心功能 |
| 音频自动播放限制 | 移动端需用户交互触发 AudioContext | 在 Sanctuary 首次触碰时解锁；提供静音备选 |
| localStorage 容量 | 90天 CrisisMarker 可能膨胀 | 定期清理旧数据；单条 marker 控制在 200B 以内 |
| 敏感表达检测准确性 | 误报导致不必要的跟进 | 宁可漏报，不可误报；跟进措辞保持通用性 |

### 伦理风险

| 风险 | 处理原则 |
|------|---------|
| 用户在危机中依赖 App 而不寻求专业帮助 | Layer 1 不是替代方案，是"等待专业帮助到来之前的缓冲"。Sanctuary 始终保留"寻求帮助"链接 |
| 敏感表达数据泄露 | 不存储具体内容，只存储布尔标记。所有对话内容留在后端 |
| 宣泄工具（沙袋）正常化暴力 | 视觉语言是"释放能量"而非"攻击"。冲击波是扩散，不是破坏 |
| 自动生成摘要不准确 | 用户必须在分享前确认/编辑。标注"AI 生成，仅供参考" |

### 产品约束

- **IGOTU 不是危机干预工具**。它是治疗间隙的陪伴工具。如果用户需要即时危机支持，应引导至专业热线。
- **IGOTU 不做临床诊断**。PHQ-9 等量表是自评工具，结果不构成医学建议。
- **所有数据导出都由用户主动触发**。系统不会自动向任何第三方发送用户数据。

## 九、设计系统扩展

### Layer 1 的视觉语言

Layer 1 的视觉应该比当前 Sanctuary 更极简：

- **背景**：纯 `var(--bg-primary)` 或更深（`#040a08`），不要渐变
- **焦点元素**：单一发光体，使用 `var(--mood-accent)` + `var(--mood-comfort-glow)`
- **动效**：比 Sanctuary 更慢（`animation-speed: 0.3`），接近呼吸频率
- **文字**：几乎没有。如果有，用 `var(--text-muted)` + 极小字号

### 新增 CSS 变量

```css
--crisis-bg: #040a08;           /* Layer 1 深底 */
--crisis-pulse-speed: 4s;       /* 脉冲速度，接近呼吸频率 */
--crisis-ripple-color: var(--mood-accent);
```

### 新增动画

| 动画名 | 用途 | 时长 | 层级 |
|--------|------|------|------|
| `punch-ripple` | 沙袋击打涟漪 | 0.4s once | L1 |
| `punch-pulse` | 中央圆脉冲 | 4s infinite | L1 |
| `mood-shift` | 累积后色调渐变 | 2s ease | L1 |
| `sound-wave` | 声景播放指示器 | 1.5s infinite | L1 |

## 十、成功指标

### 定量指标

| 指标 | 目标 | 测量方式 |
|------|------|---------|
| Layer 1 使用率（mood ≤ 2 时） | > 40% 的低谷时段至少使用一个 L1 工具 | CrisisMarker 数量 / mood ≤ 2 天数 |
| 次日回访率 | > 60% 的危机标记次日有签到 | followedUp === true 比率 |
| 情绪回升速度 | 危机后 48h 内 mood 回升 ≥ 1 分 | followUpMood - moodScore 差值 |
| 报告分享率 | > 20% 的活跃用户至少分享过一次 | 导出事件统计 |

### 定性指标

- 用户在 mood ≤ 2 时的**停留时长**是否增加（表明工具有吸引力，而非关掉 App）
- 次日签到时的**自由文本反馈**（如果未来加入）是否提到"昨天的XX有帮助"
- 治疗师对**发展摘要**格式的反馈
